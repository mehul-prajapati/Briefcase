In C, I understood that if we close a socket, it means the socket will be destroyed and can be re-used later.

How about shutdown? The description said it closes half of a duplex connection to that socket. But will that socket be destroyed like close system call?

This is explained in Beej's networking guide. shutdown is a flexible way to block communication in one or both directions. When the second parameter is 2, it will block both sending and receiving (like close). However, close is the recommended way to actually destroy a socket.

With shutdown, you will still be able to receive pending data the peer already sent (thanks to Joey Adams for noting this).

This may be platform specific, I somehow doubt it, but anyway, the best explanation I've seen is here on this msdn page where they explain about shutdown, linger options, socket closure and general connection termination sequences.

In summary, use shutdown to send a shutdown sequence at the TCP level and use close to free up the resources used by the socket data structures in your process. If you haven't issued an explicit shutdown sequence by the time you call close then one is initiated for you.

There are some limitations with close() that can be avoided if one uses shutdown() instead.

Close() will terminate both directions on a tcp connection. Sometimes you want to tell the other endpoint that you are finished with sending data, but still want to receive data.

Cloes() decrements the descriptors reference count (maintained in file table entry and counts number of descriptors currently open that are referring to a file/socket) and does not close the socket/file if the descriptor is not 0. This means that if you are forking, the cleanup happens only after reference count drops to 0. With shutdown() one can initiate normal TCP close sequence ignoring the reference count.

Parameters are as follows:

int shutdown(int s, int how); // s is socket descriptor

int how can be:

SHUT_RD or 0 Further receives are disallowed

SHUT_WR or 1 Further sends are disallowed

SHUT_RDWR or 2 Further sends and receives are disallowed

In Linux, using Unix-domain session sockets, I find that if I tell a client to Close a socket, it closes the both the socket for the connection it initiated, and also the entire socket, including the socket the server is listening on for other connections. I.e. it messes up the server. If I have my client call ShutDown(read/write), however, it does what I want and expect, which is to close down the communication socket the client had created, while leaving the listening socket up. The client or another process can then create a new socket connection to the server on the same socket address, and the socket count will have been decremented properly too.
